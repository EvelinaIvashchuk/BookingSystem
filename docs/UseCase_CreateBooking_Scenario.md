# Опис прецеденту: «Створити оренду»

---

## Загальна інформація

| Поле | Значення |
|---|---|
| **Назва прецеденту** | Створити оренду |
| **Ідентифікатор** | UC_CreateRental |
| **Актор** | Користувач (User) |
| **Зв'язані прецеденти** | <<include>> UC_ValidateRental, <<include>> UC_CheckAvailability, <<include>> UC_CheckOverlap, <<include>> UC_CalculatePrice, <<include>> UC_SendEmail |
| **Передумови** | Користувач автентифікований в системі; автомобіль знайдений та має статус Available |
| **Постумови (успіх)** | Нова оренда зі статусом `Pending` збережена в БД; розрахована вартість TotalPrice; користувач отримав email-підтвердження |
| **Постумови (відмова)** | Оренда не створена; система відображає повідомлення про помилку |

---

## Базовий сценарій (Basic Flow)

| Крок | Актор | Система |
|---|---|---|
| 1 | Користувач відкриває сторінку деталей автомобіля | |
| 2 | | Система відображає інформацію про автомобіль (марка, модель, рік, ціна за день, категорія, паливо, трансмісія) та кнопку «Орендувати» |
| 3 | Користувач натискає «Орендувати» | |
| 4 | | Система перевіряє статус автомобіля: `Available`. Відображає форму оренди з попередньо заповненими display-полями (назва, місцезнаходження, ціна за день) |
| 5 | Користувач заповнює форму: вибирає `PickupDate`, `ReturnDate`, вводить `Notes` (опційно) | |
| 6 | Користувач натискає «Підтвердити» | |
| 7 | | **UC_ValidateRental <<include>>**: FluentValidation перевіряє: PickupDate не в минулому, ReturnDate > PickupDate, тривалість 1–30 днів, не більше 60 днів наперед, Notes ≤ 500 символів. Валідація пройшла. |
| 8 | | Система перетворює ViewModel → CreateRentalDto через AutoMapper |
| 9 | | **UC_CheckAvailability <<include>>**: RentalService перевіряє, що автомобіль існує та має статус Available |
| 10 | | RentalService перевіряє: кількість активних оренд користувача < 3 |
| 11 | | **UC_CheckOverlap <<include>>**: RentalService перевіряє відсутність накладань з іншими орендами цього автомобіля на обраний діапазон дат |
| 12 | | **UC_CalculatePrice <<include>>**: TotalPrice = (ReturnDate − PickupDate).Days × car.PricePerDay |
| 13 | | Система створює об'єкт Rental зі статусом `Pending`, зберігає через `UnitOfWork.CommitAsync()` |
| 14 | | **UC_SendEmail <<include>>**: система асинхронно надсилає email-підтвердження на адресу користувача |
| 15 | | Система відображає повідомлення «Your rental has been submitted and is pending confirmation» та перенаправляє на сторінку «Мої оренди» |

## Альтернативні сценарії (Alternative Flows)

### A1 — Автомобіль недоступний (крок 4)

| Крок | Система |
|---|---|
| 4a | Система перевіряє статус автомобіля: `Rented` або `UnderMaintenance` |
| 4b | Система відображає повідомлення: *«"[Назва автомобіля]" is not available for rental»* |
| 4c | Система перенаправляє користувача на сторінку деталей автомобіля |
| | → Прецедент завершується невдачею |

---

### A2 — Невалідні дані форми (крок 7)

| Крок | Система |
|---|---|
| 7a | FluentValidation виявляє помилку (наприклад, ReturnDate < PickupDate або тривалість > 30 днів) |
| 7b | Система додає помилки до ModelState, повторно заповнює display-поля (RepopulateDisplayFields) |
| 7c | Система повертає форму з підсвіченими полями та текстом помилок |
| | → Повернення до кроку 5 |

---

### A3 — Перевищено ліміт активних оренд (крок 10)

| Крок | Система |
|---|---|
| 10a | RentalService встановлює: кількість активних оренд = 3 (максимум) |
| 10b | Система повертає ServiceResult.Fail з повідомленням: *«You already have 3 active rentals. Please cancel one before making a new reservation»* |
| 10c | Система додає повідомлення до ModelState та відображає форму з помилкою |
| | → Прецедент завершується невдачею |

---

### A4 — Накладання з існуючою орендою (крок 11)

| Крок | Система |
|---|---|
| 11a | RentalService виявляє, що обраний діапазон дат перетинається з вже існуючою підтвердженою або очікуючою орендою цього автомобіля |
| 11b | Система повертає ServiceResult.Fail: *«This car is already booked for the selected dates. Please choose different dates»* |
| 11c | Система відображає форму з повідомленням про помилку |
| | → Повернення до кроку 5 |

---

### A5 — Збій email-сповіщення (крок 14)

| Крок | Система |
|---|---|
| 14a | MockEmailService кидає виняток під час надсилання |
| 14b | SafeSendEmailAsync перехоплює виняток, записує попередження в лог (ILogger) |
| 14c | Оренда вже збережена — система продовжує нормальний потік |
| | → Продовження від кроку 15 (оренда успішна, email не надіслано) |

---

## Діаграма стану оренди

```
[Створено] → Pending
    ↓ Admin: Confirm     ↓ Admin: Reject     ↓ User: Cancel
 Confirmed             Rejected             Cancelled
```

## Бізнес-правила

| # | Правило | Де перевіряється |
|---|---|---|
| BR-1 | PickupDate не повинна бути в минулому | FluentValidation (UI) |
| BR-2 | ReturnDate > PickupDate | FluentValidation (UI) |
| BR-3 | Тривалість оренди ≥ 1 день | FluentValidation (UI) |
| BR-4 | Тривалість оренди ≤ 30 днів | FluentValidation (UI) |
| BR-5 | PickupDate ≤ UtcNow + 60 днів | FluentValidation (UI) |
| BR-6 | Notes ≤ 500 символів | FluentValidation (UI) |
| BR-7 | Автомобіль повинен мати статус Available | RentalService (бізнес-шар) |
| BR-8 | Користувач може мати максимум 3 активних оренди | RentalService (бізнес-шар) |
| BR-9 | Діапазон дат не повинен перетинатися з іншими орендами | RentalService (бізнес-шар) |
| BR-10 | TotalPrice = кількість днів × PricePerDay автомобіля | RentalService (бізнес-шар) |
