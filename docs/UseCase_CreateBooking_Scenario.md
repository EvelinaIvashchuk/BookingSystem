# Опис прецеденту: «Створити бронювання»

---

## Загальна інформація

| Поле | Значення |
|---|---|
| **Назва прецеденту** | Створити бронювання |
| **Ідентифікатор** | UC_CreateBooking |
| **Актор** | Користувач (User) |
| **Зв'язані прецеденти** | <<include>> UC_ValidateBooking, <<include>> UC_CheckAvailability, <<include>> UC_CheckOverlap, <<include>> UC_SendEmail |
| **Передумови** | Користувач автентифікований в системі; ресурс знайдений та має статус Available |
| **Постумови (успіх)** | Нове бронювання зі статусом `Pending` збережено в БД; користувач отримав email-підтвердження |
| **Постумови (відмова)** | Бронювання не створено; система відображає повідомлення про помилку |

---

## Базовий сценарій (Basic Flow)

| Крок | Актор | Система |
|---|---|---|
| 1 | Користувач відкриває сторінку деталей ресурсу | |
| 2 | | Система відображає інформацію про ресурс (назва, розташування, місткість, категорія) та кнопку «Забронювати» |
| 3 | Користувач натискає «Забронювати» | |
| 4 | | Система перевіряє статус ресурсу: `Available`. Відображає форму бронювання з попередньо заповненими display-полями (назва, розташування, місткість) |
| 5 | Користувач заповнює форму: вибирає `StartTime`, `EndTime`, вводить `Purpose` (опційно) | |
| 6 | Користувач натискає «Підтвердити» | |
| 7 | | **UC_ValidateBooking <<include>>**: FluentValidation перевіряє: StartTime > UtcNow, EndTime > StartTime, тривалість 30хв–8год, не більше 30 днів наперед, Purpose ≤ 500 символів. Валідація пройшла. |
| 8 | | Система перетворює ViewModel → CreateBookingDto через AutoMapper |
| 9 | | **UC_CheckAvailability <<include>>**: BookingService перевіряє, що ресурс існує та має статус Available |
| 10 | | BookingService перевіряє: кількість активних бронювань користувача < 3 |
| 11 | | **UC_CheckOverlap <<include>>**: BookingService перевіряє відсутність накладань з іншими бронюваннями цього ресурсу на обраний часовий слот |
| 12 | | Система створює об'єкт Booking зі статусом `Pending`, зберігає через `UnitOfWork.CommitAsync()` |
| 13 | | **UC_SendEmail <<include>>**: система асинхронно надсилає email-підтвердження на адресу користувача |
| 14 | | Система відображає повідомлення «Your booking has been submitted and is pending confirmation» та перенаправляє на сторінку «Мої бронювання» |

## Альтернативні сценарії (Alternative Flows)

### A1 — Ресурс недоступний (крок 4)

| Крок | Система |
|---|---|
| 4a | Система перевіряє статус ресурсу: `UnderMaintenance` або `Retired` |
| 4b | Система відображає повідомлення: *«"[Назва ресурсу]" is not available for booking»* |
| 4c | Система перенаправляє користувача на сторінку деталей ресурсу |
| | → Прецедент завершується невдачею |

---

### A2 — Невалідні дані форми (крок 7)

| Крок | Система |
|---|---|
| 7a | FluentValidation виявляє помилку (наприклад, EndTime < StartTime або тривалість < 30 хв) |
| 7b | Система додає помилки до ModelState, повторно заповнює display-поля (RepopulateDisplayFields) |
| 7c | Система повертає форму з підсвіченими полями та текстом помилок |
| | → Повернення до кроку 5 |

---

### A3 — Перевищено ліміт активних бронювань (крок 10)

| Крок | Система |
|---|---|
| 10a | BookingService встановлює: кількість активних бронювань = 3 (максимум) |
| 10b | Система повертає ServiceResult.Fail з повідомленням: *«You already have 3 active bookings. Please cancel one before making a new reservation»* |
| 10c | Система додає повідомлення до ModelState та відображає форму з помилкою |
| | → Прецедент завершується невдачею |

---

### A4 — Накладання з існуючим бронюванням (крок 11)

| Крок | Система |
|---|---|
| 11a | BookingService виявляє, що обраний часовий слот перетинається з вже існуючим підтвердженим або очікуючим бронюванням |
| 11b | Система повертає ServiceResult.Fail: *«This resource is already booked for the selected time slot. Please choose a different time»* |
| 11c | Система відображає форму з повідомленням про помилку |
| | → Повернення до кроку 5 |

---

### A5 — Збій email-сповіщення (крок 13)

| Крок | Система |
|---|---|
| 13a | MockEmailService кидає виняток під час надсилання |
| 13b | SafeSendEmailAsync перехоплює виняток, записує попередження в лог (ILogger) |
| 13c | Бронювання вже збережено — система продовжує нормальний потік |
| | → Продовження від кроку 14 (бронювання успішне, email не надіслано) |

---

## Діаграма стану бронювання

```
[Створено] → Pending
    ↓ Admin: Confirm     ↓ Admin: Reject     ↓ User: Cancel
 Confirmed             Rejected             Cancelled
```

## Бізнес-правила

| # | Правило | Де перевіряється |
|---|---|---|
| BR-1 | StartTime повинен бути в майбутньому | FluentValidation (UI) |
| BR-2 | EndTime > StartTime | FluentValidation (UI) |
| BR-3 | Тривалість ≥ 30 хвилин | FluentValidation (UI) |
| BR-4 | Тривалість ≤ 8 годин | FluentValidation (UI) |
| BR-5 | StartTime ≤ UtcNow + 30 днів | FluentValidation (UI) |
| BR-6 | Purpose ≤ 500 символів | FluentValidation (UI) |
| BR-7 | Ресурс повинен мати статус Available | BookingService (бізнес-шар) |
| BR-8 | Користувач може мати максимум 3 активних бронювання | BookingService (бізнес-шар) |
| BR-9 | Часовий слот не повинен перетинатися з іншими бронюваннями | BookingService (бізнес-шар) |
